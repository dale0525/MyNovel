## 1. 产品心态

MyNovel 的第一性产品心态是：**AI 小说生产线**。

它不是传统写作软件，也不是聊天式写作助手。它的核心目标是替个人作者运行一条本地 AI 网文生产线：

- 作者输入方向、偏好、禁区和审核判断。
- AI 负责规划、生成、审计、修订、状态维护和持续推进。
- 系统把需要人判断的风险推到审核台。
- 作者主要做批准、退回、要求 AI 修复，而不是逐段写正文。

产品体验上，MyNovel 应更像“生产驾驶舱 + 审核台”，而不是文字编辑器。用户看到的是当前生产阶段、下一步该确认什么、哪里有风险、AI 改了什么、是否可以放行。

## 2. 产品边界

### 必须坚持

- 个人本地使用优先。
- 全 AI 化 + 人工审核。
- 网文连载型长篇优先。
- OpenAI-compatible LLM、embedding、rerank 接口优先且唯一。
- SQLite-first，业务数据库优先 SQLite。
- 向量检索如有需要，优先本地、嵌入式、SQLite-friendly。
- 开发阶段 pixi 快捷启动。
- 正式版本优先桌面端打包。
- 界面护眼，适合长时间文字审核。
- UI 面向普通作者，不面向技术工作者。
- 开源仓库优先，默认采用 Apache-2.0 许可证。
- prompt 资产要可追踪来源、许可和版本，不能把第三方仓库 prompt 无许可混入核心代码。
- GitHub Actions 需要从第一版就预留自动测试、打包、发版和更新元数据生成流程。

### 第一阶段不做

- 重型文字编辑器。
- 多用户、登录、权限、团队协作。
- 在线发布、订阅、评论、读者数据。
- 复杂市场扫榜和平台抓取。
- 多端同步。
- 批量无人值守生产。
- 非 OpenAI-compatible SDK 的独立适配。

## 3. 目标作品形态

第一版优先服务：**网文连载型长篇**。

这个选择决定了产品优先级：

- 连续章节生产优先于文学性精修。
- 节奏、爽点、钩子、期待感优先于细腻文体打磨。
- 角色状态、伏笔账本、读者承诺、章节末尾推进感必须被持续追踪。
- 审核台要帮助用户快速判断“这一章能不能进入连载队列”。

传统长篇小说、短篇故事、纯文学改稿可以作为后续扩展，不进入 MVP 的成功标准。

## 4. 推荐路线

采用：**门控式生产线**。

MVP 先做稳定的单章流水线：

1. 开书。
2. 规划前 10 章。
3. 逐章生成。
4. 自动审计。
5. 自动修订。
6. 人工批准。
7. 更新状态。
8. 进入下一章。

最终扩展为批量生产线：用户设定“连续跑 5 章/10 章”，系统低风险自动推进，高风险暂停等待人工审核。

这条路线的判断依据：

- AI 负责生产，人负责放行，产品心态清楚。
- MVP 可以验证连续性和状态维护，不只是单章 demo。
- 不会一开始被批量并发、失败恢复和状态污染拖垮。

## 5. 用户旅程

### 5.1 开书

用户输入：

- 一句灵感。
- 题材。
- 目标读者。
- 爽点偏好。
- 写作禁区。
- 可选参考风格。

AI 输出：

- 书名候选。
- 核心卖点。
- 主角设定。
- 世界观方向。
- 核心矛盾。
- 前 10 章读者承诺。

用户只需要选择方向、确认边界、退回重做不合适的部分。

### 5.2 定盘

AI 生成初始 canon：

- 世界规则。
- 人物和关系。
- 势力与地点。
- 主线冲突。
- 关键伏笔。
- 前 10 章节奏目标。

用户审核通过后，系统锁定为生产线基础。

### 5.3 生产章节

MVP 逐章运行：

1. AI 规划本章。
2. AI 编译上下文。
3. AI 生成草稿。
4. AI 提取状态变化。
5. AI 审计。
6. AI 修订。
7. 进入人工审核。

### 5.4 人工审核

审核台展示：

- 章节正文。
- 审计问题。
- AI 修订建议。
- 状态变化。
- 影响到的角色、伏笔、地点和关系。

主操作：

- 批准。
- 退回重写。
- 让 AI 修复这些问题。
- 少量手动修正或补充审核意见。

### 5.5 状态落库

只有章节被批准后，系统才更新：

- 角色状态。
- 伏笔账本。
- 章节摘要。
- 资源与地点状态。
- 关系变化。
- 向量索引。

## 6. 核心对象

核心对象按生产线状态设计，不按文档文件夹设计。

### Book

一本书的产品定位。

字段示例：

- 题材。
- 目标读者。
- 核心卖点。
- 写作禁区。
- 风格。
- 篇幅目标。
- 连载节奏。

### Canon

被批准的事实源。

包括：

- 世界规则。
- 人物。
- 势力。
- 地点。
- 时间线。
- 资源。
- 关系。
- 伏笔。

只有通过审核的变更才能进入 canon。

### VolumePlan

卷级目标。

包括：

- 本卷核心冲突。
- 节奏曲线。
- 爽点分布。
- 关键转折。
- 阶段性承诺。

### ChapterPlan

单章生产指令。

包括：

- 章节目标。
- 必写内容。
- 禁止偏航点。
- 上下文引用。
- 字数预算。
- 结尾钩子。

### Draft

AI 生成的草稿。它不是事实源，只是候选正文。

### AuditReport

审计结果。

覆盖：

- 连续性。
- 因果链。
- 节奏。
- 角色动机。
- 信息边界。
- 伏笔。
- AI 味。
- 字数。
- 风险等级。

### StateDelta

AI 从草稿中提取的状态变化。

它必须经过校验和人工/规则批准后才能写入 canon。

### ReviewDecision

用户审核决定。

包括：

- 批准。
- 退回。
- 要求修订。
- 人工备注。

### RunTrace

每次 AI 调用和生产阶段的记录。

用于：

- 恢复。
- 调试。
- 成本统计。
- 复盘。

## 7. 关键数据边界

最重要的数据边界是：

**Draft 不可信，StateDelta 待验证，Canon 才可信。**

这条边界决定系统不会因为一次幻觉生成污染整本书。

具体规则：

- Draft 可以被反复生成和丢弃。
- StateDelta 必须结构化、可校验、可解释。
- Canon 只接受通过审核的 StateDelta。
- 向量索引是派生索引，可以重建，不能作为事实源。
- Markdown/JSON 导出是人类可读投影，不替代 SQLite 权威存储。

## 8. 生产线阶段

MVP 的章节生产线按固定阶段运行。

### Plan

根据 canon、前文摘要、卷目标和当前章节目标生成 `ChapterPlan`。

### Compose Context

从 canon、章节摘要、伏笔、角色状态和可选向量检索中选上下文，生成可审计的上下文包。

### Draft

生成章节草稿。

### Extract State

从草稿提取 `StateDelta`。

提取范围：

- 人物状态。
- 关系变化。
- 地点状态。
- 资源变化。
- 伏笔进展。
- 关键信息暴露。
- 情绪变化。

### Audit

检查：

- 连续性。
- 因果链。
- 人物动机。
- 信息边界。
- 伏笔。
- 节奏。
- 字数。
- AI 味。
- 结尾钩子。

### Revise

对可自动修复的问题进行一轮或多轮修订。

### Human Review

进入审核台。用户批准、退回、要求修复，或少量手动修改。

### Accept

批准后正文定稿，`StateDelta` 写入 canon，章节摘要和检索索引更新。

## 9. 人工审核 Gate

### 强制 Gate

- 开书定盘。
- 卷纲确认。
- 每章最终批准。
- 重大 canon 变更。
- 角色死亡/退场。
- 核心设定改写。

### 风险 Gate

- 审计出现高风险问题。
- AI 多轮修订仍失败。
- 章节偏离卷目标。
- 成本异常。
- StateDelta 校验失败。
- 连续生产中累计风险过高。

MVP 只做逐章强制批准。最终版本支持批量生产：低风险章节自动进入待审队列，高风险立即暂停。

## 10. MVP 范围

MVP 只证明一件事：

**从一句灵感出发，AI 能连续生产前 10 章可审核草稿，并且每章都能经过审计、修订、人工批准和状态落库。**

### MVP 必须包含

- 本地项目创建与模型配置。
- OpenAI-compatible LLM 调用。
- SQLite 存储。
- 开书定盘。
- 前 10 章方向生成。
- 逐章生产线：plan -> context -> draft -> extract -> audit -> revise -> review -> accept。
- 审核台：正文阅读、审计问题、状态变化、批准/退回/AI 修复。
- Canon 状态：人物、地点、势力、伏笔、章节摘要、关系/资源变化。
- RunTrace：记录每次生成、成本、错误和恢复点。
- pixi 本地启动。

### MVP 明确不包含

- 多用户/登录/权限。
- 在线发布、订阅、评论、读者数据。
- 重型文字编辑器。
- 复杂市场扫榜。
- 多端同步。
- 批量无人值守生产。
- 独立适配非 OpenAI-compatible SDK。
- 直接复制许可证不兼容的第三方 prompt。
- 应用内自动安装未经用户确认的更新。

## 11. 最终目标分层

### V1

前 10 章生产线稳定，人工逐章批准。

### V2

一卷生产，支持批量跑 N 章，高风险暂停，低风险进入待审队列。

### V3

完整连载生产，包括：

- 风格资产。
- 拆书学习。
- 长期表现分析。
- 成本策略。
- 桌面打包。
- 移动端可选。

## 12. 界面与交互原则

界面要服务“审核生产线”，不是服务“写作软件”。

### 主要页面

#### 项目首页

显示：

- 书籍状态。
- 当前生产阶段。
- 前 10 章进度。
- 待审核事项。
- 最近风险。

#### 开书定盘页

让用户用普通语言确认：

- 题材。
- 卖点。
- 主角。
- 世界观。
- 禁区。
- 前 10 章承诺。

避免展示长 prompt。

#### 章节审核台

核心页面。

布局建议：

- 左侧章节队列。
- 中间舒适阅读正文。
- 右侧审计问题和状态变化。
- 底部 AI pipeline 进度。

主要按钮：

- 批准。
- 退回。
- 让 AI 修复。

#### Canon 状态页

用卡片/表格查看：

- 人物。
- 伏笔。
- 地点。
- 势力。
- 章节摘要。
- 状态变化历史。

默认只读，必要时允许人工修正。

### 交互原则

- 默认展示“下一步该做什么”。
- 所有技术术语翻译为作者语言。
- 高风险问题要解释影响。
- 长文本阅读优先护眼：舒适行宽、字号、行距、低疲劳配色。
- 编辑能力轻量化：允许改几句、写修订指令、添加备注，不做复杂排版。
- 每次 AI 修改都要能回答：改了哪里、为什么改、影响哪些状态。
- 失败要可恢复：继续、重试、换模型、回到上一个批准点。

## 13. 技术架构

### Core

Python package，承载：

- 生产线。
- schema。
- LLM client。
- 审计。
- 状态落库。

### 环境

pixi 管理本地开发和启动命令。

### 数据库

SQLite 为权威存储。

### Schema

Pydantic/SQLModel 定义：

- Book。
- Canon。
- ChapterPlan。
- Draft。
- AuditReport。
- StateDelta。
- RunTrace。

### 模型接口

只支持 OpenAI-compatible：

- chat。
- embedding。
- rerank。

### 向量检索

MVP 可先不做，或只做 SQLite-friendly 本地索引。Canon 仍然是事实源。

### API/UI

先用本地 API + Web UI 实现审核台。后续用 Tauri/Electron 打包桌面端。

### CLI

保留开发调试入口：

- `create-book`
- `run-chapter`
- `audit-chapter`

普通用户不直接使用 CLI。

## 14. Prompt 资产治理

MyNovel 需要吸收其他优秀小说写作仓库的 prompt 思路，但要把 prompt 当作正式资产治理，而不是散落在代码里的字符串。

### 来源策略

允许吸收三类内容：

- 许可明确且兼容的 prompt 文件。
- README、论文、文档中描述的方法论。
- 从竞品/开源项目中总结出的结构化写作流程和审计维度。

不允许：

- 直接复制未声明许可证的 prompt。
- 直接复制 AGPL/GPL 项目的 prompt 到 Apache-2.0 代码库，除非单独确认许可兼容和分发边界。
- 把来源不明的 prompt 混入默认生产线。

### Prompt Registry

设计一个 prompt registry，记录：

- `id`。
- `name`。
- `purpose`。
- `version`。
- `source`。
- `source_license`。
- `adaptation_notes`。
- `model_family_hint`。
- `input_schema`。
- `output_schema`。
- `evaluation_notes`。

每个生产线阶段都通过 registry 调用 prompt：

- 开书定盘 prompt。
- 章节规划 prompt。
- 上下文编译 prompt。
- 正文生成 prompt。
- 状态提取 prompt。
- 审计 prompt。
- 修订 prompt。
- 去 AI 味 prompt。

### 迭代原则

- prompt 修改必须版本化。
- 每次生成记录 prompt id 和版本到 RunTrace。
- 默认 prompt 应偏通用和可控，题材/平台风格通过规则层叠加。
- 引入第三方 prompt 时先改写为 MyNovel 的 schema 和输出格式，再进入评估。

## 15. 开源、发版与自动更新

MyNovel 按开源仓库设计，默认采用 Apache-2.0 许可证。

### License

推荐 Apache-2.0：

- 适合工具类和桌面应用。
- 允许商业和个人使用。
- 有明确专利授权。
- 与 MIT/BSD 等宽松许可证项目较易兼容。

需要避免把 AGPL/GPL 代码或 prompt 直接混入主仓库，除非明确接受对应传染性义务或将其隔离为可选外部资源。

### GitHub Actions

规划以下 workflow：

- `ci.yml`：每次 PR/push 运行格式检查、类型检查、单元测试、基础 schema 校验。
- `docs.yml`：检查 Markdown 链接、文档结构和关键设计文档存在。
- `release.yml`：tag 触发桌面端打包、生成 changelog、创建 GitHub Release、上传安装包。
- `nightly.yml`：可选，生成开发预览包，不进入稳定更新通道。

### 版本与发布通道

采用语义化版本：

- `0.x`：设计和 MVP 阶段，允许破坏性变更。
- `1.0`：前 10 章生产线稳定，桌面端可用。
- `stable`：普通用户默认通道。
- `beta`：愿意试新功能的用户通道。

### 应用内自动更新

桌面端需要支持应用内检查更新，但更新必须可控：

- 默认只提示，不静默安装。
- 显示版本号、变更摘要、下载大小和发布时间。
- 支持跳过当前版本。
- 支持 stable/beta 通道选择。
- 更新失败必须保留当前可用版本。

如果采用 Tauri，可使用 Tauri updater 或等价机制；如果采用 Electron，可使用 electron-updater。更新元数据由 GitHub Actions 在 release 时生成并随 GitHub Release 发布。

### 发布资产

正式 release 至少包含：

- macOS arm64。
- macOS x64，如构建成本可接受。
- Windows x64。
- Linux AppImage 或 deb，作为后续目标。
- `latest.json` 或对应 updater manifest。
- checksums。
- changelog。

移动端打包不进入 MVP 发布要求。

## 17. 主要风险

### 状态污染

风险：AI 草稿中的幻觉被写入 canon，后续章节持续放大。

控制：坚持 Draft、StateDelta、Canon 三层边界；StateDelta 必须校验后写入。

### 审核负担过重

风险：AI 每章给出太多问题，用户疲劳。

控制：审计问题分级，只把高风险和会影响后文的问题放到主审核路径。

### 前 10 章质量不稳定

风险：连续章节生成容易节奏散、角色漂移。

控制：章节前做 context 编译，章节后做审计和状态落库；前 10 章作为 MVP 验收核心。

### 技术复杂度失控

风险：过早引入多端、多模型 SDK、外部向量库、批量生产。

控制：OpenAI-compatible-only、SQLite-first、pixi 本地启动、MVP 逐章批准。

### Prompt 许可污染

风险：吸收其他仓库 prompt 时引入许可证不兼容内容，影响开源发布。

控制：prompt registry 必须记录来源和许可证；未声明许可证只学习方法，不复制文本；AGPL/GPL prompt 默认不进入 Apache-2.0 主仓库。

### 自动更新误伤用户数据

风险：桌面端自动更新导致本地数据库不可用或破坏用户项目。

控制：更新前检查版本兼容和数据库迁移；迁移前备份 SQLite；默认提示更新，不静默安装。

## 18. 成功标准

MVP 成功的判断不是“能生成一章”，而是：

- 用户能从一句灵感完成开书定盘。
- 系统能连续生产前 10 章可审核草稿。
- 每章都有审计报告、StateDelta、RunTrace。
- 用户能在审核台完成批准/退回/AI 修复。
- 批准后 canon 持续更新，后续章节能引用前文状态。
- 失败时能从最近批准点恢复。
- Prompt 调用能追溯到 registry 中的 id、版本和来源。
- 仓库具备明确许可证、基础 CI 和可演进的发版/更新规划。

达到这些条件后，才进入 V2 批量生产设计。
