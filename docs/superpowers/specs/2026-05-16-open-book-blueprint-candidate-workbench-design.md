# 开书蓝图候选方向工作台设计

## 背景

当前开书蓝图页只展示候选书名、一个故事前提或核心冲突，以及读者承诺。后端和 LLM 实际保留了更多可用信息，包括题材、目标读者、卖点、主角、世界观、核心冲突、前 10 章方向，以及可选的 `candidates` 数组。作者在这个阶段的核心任务不是阅读一份完整报告，而是在多个开书方向里选出最想继续写的一版。

本次改版把蓝图页调整为“候选方向工作台”：尽可能展示 LLM 已返回的信息，同时通过信息层级、对比和折叠降低噪音。

## 目标

- 帮作者快速比较多个候选方向，而不是只阅读单份蓝图。
- 充分展示 `candidates` 和全局蓝图字段，减少“模型返回了但界面没显示”的感觉。
- 保持写作工具的安静、可扫读风格，避免把页面做成嘈杂的数据仪表盘。
- 让接受、修订、重生成等操作围绕当前候选方向发生。
- 兼容旧蓝图：没有 `candidates` 时仍能完整展示全局字段。

## 非目标

- 不在这一版引入候选评分、AI 自动评审或重新调用模型生成摘要。
- 不改变后端蓝图生成 schema 的必填字段。
- 不重做全站视觉系统。
- 不把原始 JSON 作为主界面展示。

## 页面结构

页面采用紧凑标题区加双栏工作台。

顶部标题区显示：

- `开书蓝图 vX`
- 当前状态
- 原始灵感摘要
- 如果是修订版，显示修订来源或版本提示

主工作台分为左侧内容区和右侧决策面板。

左侧内容区负责理解和比较候选方向：

- 候选方向切换标签
- 候选对比表
- 当前候选的详细蓝图

右侧决策面板负责行动：

- 当前选择
- 方向差异摘要
- 定向修订输入
- 接受前预览入口
- 接受和重生成按钮
- 原始返回和补充信息折叠区

移动端改为单列：候选切换、对比表、详情、决策面板依次向下排列。

## 候选数据归一化

前端新增 `normalizeBlueprintCandidates(content)`。

规则：

- 如果 `content.candidates` 是数组，并且 `title_options` 有可用标题，则按标题生成候选方向。
- 候选字段优先使用 candidate 自己的字段；缺失时回退到全局字段。
- 候选标题从 `title`、`selected_title`、`title_option`、`book_title` 中读取；否则按 `title_options` 同索引匹配。
- 如果没有 `candidates`，把全局字段包装成一个默认候选。
- 每个候选都保留未识别字段，用于“模型补充信息”折叠区。

候选数据形态：

- `index`
- `title`
- `genre`
- `audience`
- `sellingPoints`
- `readerPromises`
- `protagonist`
- `world`
- `centralConflict`
- `chapterDirections`
- `extras`

## 候选选择

候选方向用标签或分段控件展示。每个标签显示书名，必要时附带短题材。点击候选后：

- 左侧详情切换到该候选。
- 右侧当前选择同步更新。
- 接受按钮提交当前候选的 title。
- 修订文案默认围绕当前候选。

如果只有一个候选，仍显示候选标题，但视觉弱化为“当前方向”。

## 候选对比表

对比表用于下一版需求，本次一并实现。它解决作者在多个方向之间来回点击仍不好比较的问题。

对比表采用紧凑矩阵：

- 列：候选方向
- 行：题材、目标读者、核心冲突、主角定位、前三章钩子、主要卖点

每个单元格只显示短摘要：

- 核心冲突最多 2 行。
- 主角定位优先从 `protagonist.name`、`identity`、`role`、`goal` 提取。
- 前三章钩子从 `chapter_directions[0..2]` 的 `title` 或 `goal` 提取。
- 卖点显示前 3 个 chip。

点击对比表中的候选列可以切换当前候选。对比表不替代详情，只帮助快速判断差异。

## 候选详情

当前候选详情按“先判断，再细读”的顺序组织。

第一层：判断区

- `central_conflict` 作为主句展示。
- `genre` 和 `audience` 用短标签展示。

第二层：卖点和读者承诺

- `selling_points` 用 chip 行展示。
- `reader_promises` 用紧凑列表展示。

第三层：主角与世界观

- `protagonist` 和 `world` 并排展示。
- 对象字段优先抽取常见键：`name`、`identity`、`role`、`goal`、`flaw`、`summary`、`rules`。
- 未识别对象字段进入“更多设定”折叠区。
- 如果字段是字符串，直接作为摘要显示。

第四层：前 10 章方向

- `chapter_directions` 默认完整显示。
- 每章固定一行：章号、标题、目标。
- 支持对象数组和字符串数组。
- 对象优先展示 `title` 和 `goal`，缺失时回退到 `direction` 或字符串化摘要。

第五层：模型补充信息

- 当前候选中未识别字段统一进入折叠区。
- 全局未识别字段也可放入页面底部或右侧折叠区。
- 不直接渲染大块 JSON，优先用键值摘要；必要时再提供原始返回。

## 决策面板

右侧面板只服务“选、改、放弃这一批”。

内容顺序：

- 当前选择：候选标题、题材、目标读者。
- 方向差异摘要：从卖点、核心冲突、前三章方向抽取 2-3 个短点。
- 定向修订：textarea 文案为“想让这一批怎么改”。
- 快捷修订提示：保留当前方向加强前三章、主角更主动、融合另一个候选的世界观等。
- 接受前预览：展示接受后会固化的书名、题材、目标读者、可信设定摘要、前 10 章队列摘要。
- 主按钮：`选定这个方向，进入项目页`。
- 次按钮：`按意见重生成一版`。
- 低优先级折叠：模型原始返回、蓝图版本信息、解析错误信息。

接受前预览不阻断流程，可以是可展开区域。它的目的不是新增审核步骤，而是降低“我点接受后会发生什么”的不确定感。

## 定向修订

修订仍调用现有蓝图修订接口，但前端请求体可以附带更多上下文：

- `revisionNotes`
- `selectedTitle`
- `selectedCandidateIndex`

后端可以先忽略新增字段，保持兼容。后续如果要增强 prompt，可让修订任务明确“以当前候选为基础修改”，而不是泛泛重生成。

界面文案要引导作者写具体修改：

- 保留当前方向，但强化前三章冲突。
- 保留主角职业，换一个更强的世界规则。
- 融合另一个候选的卖点。

## 接受前预览

预览展示将被固化到项目里的内容：

- 书名
- 题材
- 目标读者
- 核心冲突
- 主角摘要
- 世界观摘要
- 前 10 章标题或目标

该预览从当前候选数据生成，不再请求 LLM。后端已有按选中书名取 candidate 内容创建书、Canon、章节和卷计划的能力，因此预览应和接受后的数据来源保持一致。

## 状态与错误

生成中、失败、重试和 pending 操作沿用现有等待状态。

失败状态仍显示：

- 错误消息
- 解析错误
- 重试按钮

成功状态下：

- 没有候选时显示“当前蓝图没有可用候选方向”，但仍显示原始返回折叠区。
- 有标题但字段缺失时，显示可读的空状态，不显示 `undefined` 或空 JSON。
- 所有异步按钮在 pending 时禁用，继续防重复提交。

## 视觉原则

- 采用现有工作台风格：暖纸背景、低饱和棕绿、轻边框、克制阴影。
- 信息密度提升，但每块只承担一个判断任务。
- 标签和 chip 用于扫读，不滥用徽章。
- 不使用大面积渐变、装饰图形或营销式 hero。
- 按钮和输入控件保持现有样式。
- 固定格式区域使用稳定网格和最小高度，避免切换候选时大幅跳动。

## 测试计划

前端测试覆盖：

- 有 `candidates` 时渲染多个候选标签。
- 点击候选后详情和右侧当前选择同步更新。
- 对比表显示题材、目标读者、核心冲突、主角定位、前三章钩子和主要卖点。
- 无 `candidates` 的旧蓝图仍显示全局字段。
- `selling_points`、`reader_promises`、`protagonist`、`world`、`chapter_directions` 都实际出现在页面中。
- 接受按钮提交当前候选标题。
- 修订请求携带 `revisionNotes`，并可携带当前候选上下文。
- pending 状态仍防重复点击。

后端测试如需增强：

- 接受选中书名时继续使用对应 candidate 内容。
- 新增修订上下文字段时保持旧请求兼容。

## 实施顺序

1. 在前端蓝图页附近增加候选归一化和字段摘要 helper。
2. 重构 `BlueprintPage` 成候选工作台结构。
3. 实现候选标签、对比表、详情模块、决策面板和接受前预览。
4. 补充样式，保持现有工作台视觉体系。
5. 更新前端测试。
6. 如需要，扩展修订 API 请求体测试，但不强制改变后端行为。
